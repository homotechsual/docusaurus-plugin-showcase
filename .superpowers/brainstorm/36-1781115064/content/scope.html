How much of the plugin config should the playground cover?

The showcase plugin's config has two tiers: simple scalar options, and complex tag/status object definitions. How deep should the playground go?

plugins: [
  '@homotechsual/docusaurus-plugin-showcase',
  pluginsPreset({
    routeBasePath: 'plugins',
    dataDir: './data/plugins',
    pageTitle: 'Plugin Directory',
    screenshotUrl: 'https://...'
  })
]

A — Preset-focused

Users pick pluginsPreset or sitesPreset (or neither), then configure the scalar options: routeBasePath, dataDir, pageTitle, pageDescription, submitUrl, screenshotUrl, favouriteTag. Tag and status definitions come pre-baked from the chosen preset.

Pros

  • Covers the 90% case cleanly
  • Simple, readable UI
  • Generated config is concise

Cons

  • Doesn't help custom-tags users
plugins: [['showcase', {
  routeBasePath: 'tools',
  tags: {
    react: {
      label: 'React',
      description: 'React-based',
      color: '#61dafb'
    }
  },
  dataDir: './data/tools'
}]]

B — Full options

Covers everything: scalar options plus a dynamic tag/status builder where users can add, name, colour, and icon each tag. Generates the full raw plugin config including the tags object.

Pros

  • Useful even without presets
  • Shows all possibilities

Cons

  • Complex UI (add/remove rows)
  • Colour picker needed
  • Much larger generated output
// Preset mode
...pluginsPreset({ routeBasePath: 'plugins' }),

// Custom mode (advanced)
tags: { react: { label: 'React', color: '#61dafb' } }

C — Preset + Custom toggle

A toggle switches between Preset mode (simple, pick pluginsPreset or sitesPreset + scalar options) and Custom mode (advanced, exposes the full tag/status builder). Both generate appropriate output for their mode.

Pros

  • Best of both worlds
  • Progressive disclosure

Cons

  • More complex to build
  • Two modes to maintain
